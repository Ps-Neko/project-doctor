# 외부 검증 코퍼스 (external-corpus)

> **목적**: 내부 픽스처(자기 표본)를 넘어, **남이 만든 실제 프로젝트**로 검진 품질을 측정한다.
> 지금 진단 품질의 가장 큰 공백은 "실사용 n=1·자기 프로젝트"다(BACKLOG **BL-14**). 이 폴더는
> 외부 사용자가 생기면 그 프로젝트(익명화)로 정답지를 만들어 채우는 **틀**이다. 지금은 비어 있다.

## 왜 필요한가

내부 픽스처(`tests/fixtures/`)는 "문제를 일부러 심은" 표본이라, 스킬이 그 문제를 찾도록
이미 맞춰져 있을 수 있다(자기 채점 편향). 외부의 진짜 프로젝트는 그런 편향이 없어,
탐지율이 실제로 일반화되는지·오탐이 늘지 않는지를 정직하게 보여준다.

## 목표 구성 (예시 — 형편껏)

| 종류 | 권장 수 | 무엇을 보나 |
|------|--------|------------|
| 소형 Python | 5 | 중복·죽은코드·하드코딩 일반화 |
| 소형 JS/TS | 5 | 언어 무관 탐지 |
| 문서 없는 인수인계 | 3 | `intro`(초진) 품질 |
| 공개 전(leaky) 샘플 | 3 | 비밀키·개인정보 탐지 일반화 |
| AI 생성 messy | 5 | 주 사용 시나리오 |

## 한 샘플 추가하는 법

1. **허가**: 타인의 코드면 공유·익명화 허가를 먼저 받는다.
2. **익명화**: 실제 비밀키·개인정보·회사명·실명을 가짜로 치환한다(코드 구조는 보존).
   - ⚠️ 실제 비밀키가 남으면 안 된다 — git 과거 기록까지 점검한다.
3. 폴더를 만든다: `external-corpus/<별칭>/` (별칭은 출처를 숨긴 임의 이름, 예: `corp-a-billing`).
4. 익명화한 프로젝트 코드를 그 폴더에 둔다(코드는 git 에 커밋되지 않는다 — 아래 "커밋 정책").
5. `_TEMPLATE/EXPECTED.md`를 복사해 `<별칭>/EXPECTED.md`로 정답지를 만든다.
   심은 문제가 아니라 **사람이 코드를 읽고 합의한 실제 문제**를 적는다.
6. `_TEMPLATE/NOTES.md`를 복사해 `<별칭>/NOTES.md`에 출처·익명화 내역·검진 환경(모델/날짜)을 적는다.

## 채점

내부 픽스처와 같은 도구를 쓴다(저장소 루트에서):

```bash
python tests/run_checks.py "<검진 보고서.md>" "tests/external-corpus/<별칭>/EXPECTED.md"
```

- 형식 계약 통과 (`verify_report_format.py`)
- 탐지율 ≥ 80% + 오탐 0 (`compare_report.py`)
- 위치 오진 0 (`check_location_sampling.py` · BL-22)

여러 샘플을 모으면, **내부 픽스처 점수와 외부 점수의 격차**가 곧 일반화 여부의 증거다.

## 커밋 정책

- **외부 프로젝트 코드 자체는 커밋하지 않는다**(타인 자산·민감 가능). 이 폴더의 `.gitignore`가
  코드를 제외하고, 정답지(`EXPECTED.md`)·메모(`NOTES.md`)·이 틀(`README.md`·`_TEMPLATE/`)만 추적한다.
- 정답지·메모에도 익명 별칭만 쓴다. 실명·회사명·실제 키를 적지 않는다.
