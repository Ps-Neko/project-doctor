# 사례 1 — 검진 → 치료 → 재검진: 바이브코딩 도구 실치료기 (D→C)

> 실제 사용자(이 스킬의 제작자)가 자기 업무 도구를 검진·치료한 기록입니다. 모든 보고서는 스킬 v1.2.0이 실제로 출력한 측정 산출물이며, 환자 프로젝트 소유자의 동의(본인) 하에 공개합니다.

## 환자 소개

AI와 함께 만든 실무 계산 도구(React 웹앱 — 리액터 프레싱블럭 위치 선정). 증상은 비개발자 프로젝트의 전형이었습니다:

- 같은 도구의 **옛 버전 HTML 파일 16개**가 폴더에 그대로 (`1.0ver` ~ `5.0ver`, 그중 2개는 내용까지 같은 쌍둥이)
- 프로젝트 전체의 **완전 중복 사본 폴더**가 옆에 하나 더 + zip 압축본까지
- README는 개발 도구가 만들어 준 견본 그대로, 테스트 0건, git 없음 — "복사해서 새 버전 만들기"로 버텨 온 상태

## 첫 검진 — 종합 D (그런데 심각은 0건)

```
부위별: 구조 C · 정리 C · 문서 C · 테스트 B · 안전 C
종합: 🔴 D — 치료가 필요해요 (합산 6.0점)
발견: 심각 0 · 권장 5 · 참고 2
```

부위는 전부 B~C인데 종합이 D — **자잘한 문제 7건이 합산되어 등급을 끌어내린** 경우입니다. 결과지의 "종합은 부위 평균이 아니라 합산 점수" 설명이 정확히 이 상황을 위해 있습니다.

## 치료 — 승인한 것만, 전부 되돌릴 수 있게

사용자가 보고서에서 항목을 고르면, 한 번에 하나씩 처리하고 항목마다 커밋(되돌리기 지점)을 남깁니다. git이 없던 폴더라 **치료 첫 단계가 "git 켜기 + 실행 전 체크포인트"**였습니다.

| 치료 | 내용 |
|---|---|
| 옛 버전 정리 | 19개 파일을 보관함으로 이동 → 이후 git이 이력을 보존하므로 보관함 자체도 정리 |
| README 2개 | "현행본은 무엇인가 + 어떻게 켜는가"를 실제 내용으로 |
| 자동 검사 신설 | 계산 엔진 테스트 25건 — 외부 설치 없이 `npm test`로 동작 |
| 숫자에 이름표 | 합격 판정 범위(45~50°)·도면 크기 등을 상수 한 곳으로 |
| 이식성 | 특정 PC 전용 절대 경로 제거, 미사용 의존성 제거 |

치료 커밋 14개 — 어느 시점으로든 한 줄로 되돌릴 수 있습니다.

## 최종 재검진 — C (부위는 A 3개 + B 2개)

```
부위별: 구조 B · 정리 A · 문서 A · 테스트 A · 안전 B
종합: 🟡 C — 주의가 필요해요 (합산 2.5점)
남은 발견: 권장 2 (거대 함수 4개 · CDN 의존) · 참고 1
```

문서·테스트·정리가 A로 올라섰고, 남은 것은 사용자가 의도적으로 보류한 큰 수술(함수 분리)과 오프라인 대응(CDN 번들화)뿐입니다.

## 정직한 한계 — 이 사례에서 도구가 배운 것

같은 환자를 **이전 기억이 없는 검진자(새 세션) 5명**이 검진했더니, 큰 발견은 일치했지만 매번 조금씩 다른 "이름 없는 숫자(매직 넘버)" 꼬리를 새로 찾았습니다. 계산식 → 도면 → 모드 문자열 → 캔버스 크기 → 라벨 문구 순으로요.

- 이것이 **주치의가 한 번의 검진으로 모든 것을 찾는다고 약속하지 않는 이유**이고, 보고서가 발견마다 ID와 파일·줄 위치를 박아 사람이 직접 확인할 수 있게 만드는 이유입니다.
- 또한 **정기 재검진이 설계의 일부인 이유**이기도 합니다 — 실제 건강검진이 그렇듯, 한 번에 다 보이지 않습니다.

등급도 솔직히 말하면: 합산 점수가 경계(2.5점)에 있어, 검진자에 따라 C와 B 사이에서 흔들릴 수 있는 상태입니다. 그래서 이 사례의 결론은 "D→B 성공!"이 아니라 **"D→C, 부위 올 A·B, 그리고 도구의 한계까지 그대로"**입니다.
