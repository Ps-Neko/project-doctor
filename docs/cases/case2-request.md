# 사례 2 — 은퇴한 전설의 부검: npm `request` 검진기

> 이 문서는 **평가가 아니라 도구 체험기**입니다. `request`는 한때 npm에서 가장 많이 쓰인 HTTP 라이브러리였고, 2020년에 **스스로 지원 종료(deprecated)를 선언하고 명예롭게 은퇴**한 프로젝트입니다. 은퇴를 비판할 이유는 없습니다 — 다만 "은퇴 당시의 코드 건강 상태"를 주치의가 어떻게 읽어내는지 보여드리기에 이보다 좋은 환자가 없을 뿐입니다.
>
> ⚠️ **면책**: 아래는 자동 진단 도구의 출력이며 오진 가능성이 있습니다. 모든 발견은 게시 전에 사람이 파일·줄 단위로 재검증했지만, 원 프로젝트 관리자가 원하시면 이 문서를 즉시 내리겠습니다.

## 먼저, 잘하고 있던 것들

검진 결과에서 가장 먼저 보이는 것은 **모범적인 부위 3개**입니다:

- **문서 A** — README·사용법·부품 목록이 완비
- **테스트 A** — 자동 확인 장치(테스트 파일)가 55개
- **안전 A** — 특정 컴퓨터 전용 경로·백업 잔재 없음

전성기에 수천만 프로젝트가 의존했던 라이브러리답습니다.

## 그리고, 은퇴의 이유가 보이는 한 부위

**구조 D** — 핵심 로직이 `request.js` **한 파일 1,553줄**에 몰려 있고, 그 안의 함수 하나(`init`)가 **443줄**입니다. "고치기가 점점 어려워져 유지보수 부담이 한계에 달했다"는 은퇴 선언문의 사정이, 검진표의 구조 부위에 그대로 찍혀 있는 셈입니다.

검진 과정에서 도구의 안전장치들도 실전 동작을 확인했습니다: 파일이 120개라 **규모 가드**(100개 초과 시 핵심 항목 위주로 축소 제안)가 발동했고, 테스트용 인증서·키 파일 19개는 **보안 파일 열람 금지 규칙**에 따라 내용을 열지 않고 존재만 기록했습니다.

## 이 사례가 보여주는 것

진단 도구는 "나쁜 프로젝트"를 찾는 게 아닙니다. `request`는 문서·테스트·안전이 A인 훌륭한 프로젝트였고, 그럼에도 **구조 한 부위의 부채가 유지보수를 무너뜨릴 수 있다**는 것 — 그리고 그것이 검진표 한 장에서 5초 만에 보인다는 것이 핵심입니다. 당신의 프로젝트가 어느 부위부터 무거워지고 있는지, 늦기 전에 확인해 보세요.
