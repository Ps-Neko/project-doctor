# 자주 묻는 질문 (FAQ)

## "그냥 Claude한테 '내 프로젝트 점검해줘'라고 하면 되잖아요?"

됩니다 — 그리고 꽤 괜찮은 답을 받을 겁니다. 차이는 **그때그때 다른 친절한 답**과 **측정된 절차** 사이에 있습니다:

| | 맨몸 Claude에게 부탁 | 프로젝트 주치의 |
|---|---|---|
| 점검 항목 | 그날그날 다름 | **표준 카탈로그 35종 ID** — 같은 문제는 항상 같은 이름으로 |
| 결과 검증 | 불가 (매번 다른 산문) | 발견마다 **파일:줄 위치 의무** + 정답지 대비 **탐지율 3회 연속 100% 측정** ([EVALS](../EVALS.md)) |
| 다음 검진과 비교 | 불가 | ID가 고정되어 있어 "지난번보다 좋아졌나"를 기계적으로 비교 |
| 수정 안전 | 부탁하기 나름 | **진단은 읽기 전용** · 승인한 항목만 · 항목마다 **되돌리기 한 줄** 보장 |
| 보고서 | 개발자 용어 섞임 | 항목마다 "무슨 뜻인가요? / 어디? / 고치면? / 승인 명령" 4단 — 비개발자 기준 |

린터(ESLint 등)·AI 코드리뷰 도구와는 보는 곳이 다릅니다 — 그것들은 **코드 문법·버그**를 보고, 주치의는 **프로젝트 운영 건강**(옛 버전 사본 방치, 설명서 부재, 인수인계 불가능, 공개 전 비밀키)을 봅니다. 일부 항목(함수 길이·매직 넘버)은 린터도 별도 규칙을 켜면 잡을 수 있지만 — 사례 1에서 그 프로젝트에 이미 설치돼 있던 ESLint 기본 설정이 잡아준 것은 7건 중 0건이었고, "옛 버전 16개 방치"나 "README가 견본 그대로" 같은 발견은 애초에 린터의 영역이 아닙니다.

## 진단이 틀릴 수도 있나요?

있습니다 — 정직하게 말씀드립니다. [사례 1](./cases/case1-reactor.md)에서 새 세션의 검진자 5명이 큰 발견은 일치했지만 잔잔한 발견(이름 없는 숫자)은 매번 조금 다르게 찾았습니다. 그래서:

- 모든 발견에 **ID와 파일:줄 위치**를 박아 사람이 직접 확인할 수 있게 하고,
- 확신이 부족한 관찰은 발견에 싣지 않고 별도 절로 분리하며,
- 한 번의 검진이 아니라 **정기 재검진**을 설계의 일부로 둡니다.

오진을 발견하시면 이슈로 알려주세요 — 카탈로그 기준을 다듬는 가장 좋은 재료입니다.

## 내 프로젝트 내용이 어디로 가나요?

검진 시 프로젝트 내용이 **Claude(Anthropic) 서버로 전송됩니다** — 이 스킬이 아니라 Claude Code 자체의 기본 동작입니다. 회사 기밀·고객 데이터가 있다면 회사의 AI 사용 정책을 먼저 확인하세요. 이 스킬이 별도의 제3 서비스로 내용을 보내는 일은 없습니다.

## 치료(자동 수정)가 무섭지 않을까요?

치료의 3원칙이 있습니다: ① 보고서가 나오기 전에는 **아무것도 고치지 않음** ② 사용자가 **승인한 항목만, 한 번에 하나씩** ③ 실행 전 체크포인트(git 또는 백업)가 없으면 **실행 자체를 거부**하고, 항목마다 되돌리기 명령 한 줄을 인쇄합니다. 사례 1에서 치료 14건 전부가 이 절차로 진행됐습니다.

## 비용은요?

검진은 Claude 사용량(토큰)을 소모합니다. 작은 프로젝트 기본 검진은 가볍고, 정밀 검진(`--deep`)은 몇 배 더 듭니다. 파일 100개가 넘으면 규모 가드가 "축소해서 볼까요?"라고 먼저 물어봅니다.

## 큰 프로젝트·다른 언어도 되나요?

카탈로그 항목은 언어 중립(파일·구조·문서 기준)이라 대부분 동작하지만, 측정(탐지율 100%)은 내부 픽스처 기준입니다 — 실제 프로젝트에서의 성능을 보증하지 않습니다 ([EVALS의 한계 절](../EVALS.md) 참조). 수십만 줄 모노레포보다는 개인·소규모 프로젝트에 맞춰 설계됐습니다.
