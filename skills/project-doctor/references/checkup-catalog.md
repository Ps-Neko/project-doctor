# 진단 항목 카탈로그 (checkup-catalog)

> 건강검진 모드의 진단 기준. 발견 항목의 ID·탐지 방법·심각도 판정·쉬운 설명이 모두 여기서 나온다.
> "감"으로 진단하지 않는다 — 이 카탈로그에 있는 항목만, 여기 적힌 기준으로 판정한다.

## ID 계약 (절대 규칙)

1. **ID 불변**: 한번 부여된 ID(예: `DUP-01`)의 의미는 절대 바뀌지 않는다.
2. **결번 유지**: 항목을 폐기해도 번호를 당겨 쓰지 않는다 (폐기 표시만 남긴다).
3. **재사용 금지**: 폐기된 ID를 다른 의미로 다시 쓰지 않는다.
4. **추가 규칙**: 새 항목은 해당 카테고리의 마지막 번호 + 1로만 추가한다.
5. **동시 갱신**: 항목을 추가·수정하면 같은 작업에서 ① 픽스처에 문제 심기 ② EXPECTED.md 갱신 ③ 대조 스크립트 재실행까지 한 세트로 끝낸다.

## 코어 항목 본문 (기본 검진 첫 묶음 — 11개)

각 항목: **탐지 방법** (Grep/Glob 수준으로 구체화) / **심각도 판정** / **쉬운 설명 템플릿**.
보고서 작성 시 쉬운 설명 템플릿을 그대로 쓰되 프로젝트 상황에 맞게 빈칸만 채운다.

### DUP-01 — 같은 코드 블록이 여러 곳에 복사됨
- 탐지: 함수 정의들을 훑어 이름·본문이 사실상 동일한(6줄 이상 일치) 블록을 찾는다. 특징적인 줄(고유한 수식·문자열)을 Grep으로 전체 검색해 다른 파일에서 같은 블록이 나오는지 확인.
- 판정: 3곳 이상 또는 돈·계산 관련 핵심 로직이면 🔴 심각, 2곳이면 🟡 권장.
- 쉬운 설명: "같은 기능의 코드가 N곳에 복사되어 있습니다. 한 곳을 고치면 나머지도 따로 고쳐야 해서, 빠뜨리면 버그가 됩니다."

### DEAD-02 — 주석 처리된 코드 덩어리
- 탐지: 연속 10줄 이상이 주석(`#`, `//`)인데 내용이 문장이 아니라 코드 형태(`=`, `()`, `if` 등 포함)인 구간. Grep: `^\s*(#|//)` 연속 구간 확인.
- 판정: 10~29줄 ⚪ 참고, 30줄 이상 또는 3곳 이상이면 🟡 권장.
- 쉬운 설명: "예전 코드를 지우는 대신 주석(메모)으로 남겨둔 덩어리입니다. 읽는 사람이 '이거 살아있는 코드인가?' 헷갈리게 만듭니다. 버전 관리(git)가 있다면 지워도 언제든 되찾을 수 있습니다."

### BIG-01 — 거대 파일
- 탐지: 코드 파일의 줄 수 측정 (제외 목록 적용 후). 임계값: 800줄.
- 판정: 800줄 초과 🟡 권장, 1,500줄 초과 🔴 심각. (단, 내용의 약 90% 이상이 데이터 목록이고 로직 함수가 없는 "순수 데이터 파일"일 때만 한 단계 낮춤 — 데이터와 함수가 섞여 있으면 낮추지 않음)
- 쉬운 설명: "파일 하나가 N줄입니다. 책으로 치면 목차 없는 N쪽짜리 챕터라, 원하는 부분을 찾기도 고치기도 어렵습니다. 역할별로 나누는 것을 권합니다."

### BIG-02 — 거대 함수
- 탐지: 함수 정의(`def`, `function`, 화살표 함수)의 줄 수 측정 — 함수 첫 줄부터 본문 마지막 줄까지, 빈 줄·주석 포함. 임계값: 50줄.
- 판정: 50~99줄 ⚪ 참고, 100줄 초과 🟡 권장.
- 쉬운 설명: "함수 하나가 N줄입니다. 함수는 '한 가지 일'을 하는 단위인데, 이 길이면 여러 일이 섞여 있을 가능성이 큽니다."

### HARD-01 — 하드코딩된 절대 경로
- 탐지: Grep 패턴 — `[A-Za-z]:\\` 또는 `[A-Za-z]:/`로 시작하는 경로 문자열, `/Users/`, `/home/`. (제외 3종: 주석 속 예시 / URL 내부 `://` 뒤 문자열 — `https://` 오탐 방지 / 문서(.md) 속 설명·예시로만 등장하는 경로)
- 판정: 실행 경로에 영향을 주면 🔴 심각 (다른 컴퓨터에서 100% 깨짐).
- 쉬운 설명: "특정 컴퓨터에만 있는 폴더 주소(예: C:\\Users\\○○\\...)가 코드에 박혀 있습니다. 다른 컴퓨터에서는 그 주소가 없어서 바로 오류가 납니다."

### DOC-01 — README 부재
- 탐지: Glob `README*` 0건.
- 판정: 🔴 심각 (이 프로젝트가 뭔지 알 방법이 코드뿐).
- 쉬운 설명: "프로젝트 설명서(README)가 없습니다. 다른 사람(그리고 6개월 뒤의 나)은 이게 뭐 하는 프로젝트인지, 어떻게 켜는지 알 수 없습니다."

### DOC-02 — README에 실행 방법 없음
- 탐지: README가 있을 때, 실행·설치 관련 표현(실행, 설치, run, install, usage, 사용법, `python`, `npm` 등)이 한 줄도 없는지 확인.
- 판정: 🟡 권장. (README 자체가 없으면 DOC-01만 보고하고 이 항목은 생략)
- 쉬운 설명: "설명서는 있는데 '어떻게 켜는지'가 없습니다. 처음 받은 사람은 결국 코드를 열어봐야 합니다."

### DOC-03 — 의존성 목록 파일 부재
- 탐지: Python 파일에 표준 라이브러리가 아닌 import(예: requests, pandas)가 있는데 `requirements.txt`·`pyproject.toml`이 없음 / JS 파일에 require·import가 있는데 `package.json`이 없음.
- 판정: 🟡 권장.
- 쉬운 설명: "이 프로젝트가 어떤 부품(외부 패키지)을 쓰는지 적은 목록이 없습니다. 새 컴퓨터에서 돌리려면 무엇을 설치해야 하는지 시행착오로 알아내야 합니다."

### TEST-01 — 테스트 전무
- 탐지: Glob — `tests/`, `test_*.py`, `*_test.py`, `*.test.js`, `*.spec.js` 전부 0건.
- 판정: 🟡 권장.
- 쉬운 설명: "고친 뒤에 '아직 잘 도는지' 자동으로 확인해 주는 안전장치(테스트)가 없습니다. 지금은 고칠 때마다 사람이 일일이 눌러봐야 합니다."

### TMP-01 — 임시·백업 파일 잔재
- 탐지: Glob — `*.bak`, `*~`, `*.old`, `*backup*`, `*copy*`, `*복사본*`.
  - **버전 사본 더미**도 포함: 같은 기반 이름에 버전 번호만 다른 사본이 3개 이상이면(예: `도구 1.0ver.html`·`도구 2.0ver.html`·`도구 3.0ver.html`, `report_v1`·`report_v2`…) 하나의 TMP-01로 묶어 보고한다(파일마다 따로 세지 않는다).
- 판정: 🟡 권장 (혼란 유발 + 옛 코드를 실수로 고칠 위험).
- **보관함 흡수**: 이런 사본들이 한 정리용 폴더(`archive/`·`old/`·`backup/`·`_old/` 등)에 모여 있으면 TMP-01로 **한 번만** 보고하고, 그 폴더 내부 파일은 DUP-01·BIG-01·STRUCT 등 다른 ID로 **중복 집계하지 않는다**(보관 의도가 분명한 폴더이므로 — 같은 원인 이중 보고 금지, HARD-02 흡수와 동일 원칙). 단 비밀키·개인정보(SEC·PII·HARD-01)는 보관함 안이라도 보고한다.
- 쉬운 설명: "'예전 버전 백업'으로 보이는 파일이 남아 있습니다. 진짜 코드와 헷갈려서 엉뚱한 파일을 고치게 되는 흔한 원인입니다."

### STALE-02 — 버려진 실험 파일
- 탐지: 이름이 실험성(`test123`, `tmp`, `scratch`, `untitled`, `new1`, `테스트용` 등)이고, 다른 파일 어디에서도 import·참조되지 않는 파일.
- 판정: 🟡 권장.
- 쉬운 설명: "실험하다 두고 간 것으로 보이는 파일입니다. 프로젝트의 진짜 부품이 아니라면 정리(보관 후 삭제)를 권합니다."

## 확장 항목 본문 (기본 검진 둘째 묶음 — 11개)

### DUP-02 — 거의 같은 함수가 이름만 다르게 존재
- 탐지: 함수 본문이 80% 이상 유사(변수명·사소한 값만 다름)한 6줄 이상 함수 쌍.
- 판정: 🟡 권장. / 쉬운 설명: "이름만 다르고 하는 일이 거의 같은 함수가 2개 있습니다. 어느 쪽을 써야 할지 헷갈리고, 한쪽만 고쳐지는 사고가 납니다."

### DEAD-01 — 어디서도 쓰지 않는 함수·클래스
- 탐지: 정의된 함수·클래스 이름을 Grep으로 전체 검색해 정의 외 참조가 0건. (외부에 공개하는 라이브러리 함수일 가능성이 있으면 보고하지 않음 — 보수적)
- 판정: 🟡 권장. / 쉬운 설명: "만들어 두고 아무도 부르지 않는 기능입니다. 읽는 사람의 시간을 뺏고, '이거 지워도 되나?'라는 고민을 남깁니다."

### DEAD-03 — 어디서도 불러오지 않는 파일
- 탐지: 코드 파일인데 import·require·`<script src>` 등 참조가 0건. 엔트리포인트(main, index, app 등)는 제외. **참조 방식을 확신할 수 없는 구조면 보고하지 않는다** (보수적 — 오탐 방지). 실험성 이름이면 STALE-02로만 보고 (중복 보고 금지).
- 판정: 🟡 권장. / 쉬운 설명: "프로젝트 어디에서도 사용하지 않는 파일로 보입니다."

### BIG-03 — 깊은 중첩
- 탐지: 들여쓰기 4단계를 초과하는 블록 (if 안의 for 안의 if 안의 ...).
- 판정: ⚪ 참고. / 쉬운 설명: "코드가 4겹 이상 안으로 파고들어 있습니다. 읽다가 길을 잃기 쉬운 구조입니다."

### HARD-02 — 반복되는 매직 넘버·문자열
- 탐지: **같은 의미의 값이 3회 이상 반복**될 때만 본다 (검진자 간 일관성을 위해 대상·제외를 명확히 한다).
  - **대상** (이름을 붙이면 뜻이 분명해지는 값): 요율·한도·임계·기준값 같은 비즈니스/설정 값 (예: 수수료율 `0.05`, 한도 `50000`, 합격 각도 `45`).
  - **제외** (이름을 붙여도 의미가 늘지 않는 값): `0`·`1`·`-1`, 배열 인덱스, 화면 좌표·크기·여백 같은 구조적 수치, HTTP 상태 코드, 흔한 단위 환산값.
  - **흡수 규칙**: 반복의 원인이 중복 코드 블록(DUP-01)이면 DUP-01로 흡수하고 따로 보고하지 않는다 (같은 원인 이중 보고 금지 — DOC-03/DEP-02 관계와 동일).
- 판정: ⚪ 참고. / 쉬운 설명: "의미 있는 값(예: 수수료율 0.05)이 이름 없이 여기저기 박혀 있습니다. 값이 바뀌면 전부 찾아 고쳐야 합니다."

### HARD-03 — 하드코딩된 URL·IP
- 탐지: Grep — `http(s)://`, IP 주소 패턴. `localhost`·`example.com`·공식 문서 URL은 제외.
- 판정: 🟡 권장 (서버가 바뀌면 코드를 고쳐야 함). / 쉬운 설명: "서버 주소가 코드에 직접 박혀 있습니다. 주소가 바뀌면 코드 곳곳을 고쳐야 하고, 내부 주소가 외부에 노출될 수도 있습니다."

### TMP-02 — 빌드 산출물이 소스에 섞임
- 탐지: `__pycache__`, `*.pyc`, `dist/`, `build/` 등이 소스와 같은 위치에 존재 (Glob — 분석 제외 목록과 별개로 존재 여부만 보고).
- 판정: ⚪ 참고. / 쉬운 설명: "자동으로 다시 만들어지는 부산물이 소스 코드와 섞여 있습니다. 지워도 되는 파일들입니다."

### STALE-01 — 화석화된 TODO·FIXME 다수
- 탐지: Grep — `TODO`, `FIXME`, `XXX` 합계 5건 이상.
- 판정: ⚪ 참고. / 쉬운 설명: "'나중에 고치자'는 메모가 N건 쌓여 있습니다. 메모가 많아지면 아무도 안 보게 됩니다 — 할 일 목록으로 옮기거나 지우는 게 낫습니다."

### STRUCT-01 — 폴더 구조 없음
- 탐지: 코드 파일 15개 이상이 전부 한 폴더에 있고 하위 폴더 구분이 없음.
- 판정: 🟡 권장. / 쉬운 설명: "서랍 없이 책상 위에 서류 N장이 다 펼쳐져 있는 상태입니다. 역할별 폴더로 나누면 찾기 쉬워집니다."

### STRUCT-02 — 같은 역할 파일이 흩어져 있음
- 탐지: 같은 역할(이름 패턴·내용)의 파일이 3개 이상 폴더에 분산. 확신이 없으면 보고하지 않음 (보수적).
- 판정: ⚪ 참고. / 쉬운 설명: "같은 종류의 파일이 여러 폴더에 흩어져 있어 한눈에 모이지 않습니다."

### STRUCT-03 — 이름과 내용 불일치
- 탐지: 파일·폴더 이름이 약속하는 것과 실제 내용이 명백히 다른 경우만 (예: `utils`에 프로그램의 시작점이 들어 있음).
- 판정: ⚪ 참고. / 쉬운 설명: "라벨과 내용물이 다른 서랍입니다. 찾는 사람이 엉뚱한 곳을 뒤지게 됩니다."

## 정밀 검진 절 (--deep 전용)

> 시작 전 고지: "정밀 검진은 기본 검진보다 시간과 사용량(토큰)이 몇 배 더 듭니다." 진행 확인 후 수행.
> 기본 검진 전체 + 아래 4가지를 추가로 수행한다. 고지·진행 확인·AI 문진(④)은 가능한 한 한 메시지로 묶어 질문 횟수를 최소화한다.

### ① 병력 청취 — HIST-01 (git 이력 핫스팟)
- 탐지: git이 있으면 `git log --name-only`로 파일별 수정 횟수 집계 (읽기 전용). "수정 횟수" = **그 파일을 변경한 커밋 수 (최초 추가 커밋 포함)**. 수정 횟수 상위이면서 크기·복잡도도 큰 파일 = 핫스팟.
- 판정: 🟡 권장. 보고서 근거에 "최근 N회 수정됨"을 반드시 부착. git이 없으면 이 절은 생략하고 그 사실만 한 줄 보고.
- 쉬운 설명: "가장 자주 고치는데 가장 큰 파일입니다 — 문제가 생길 확률이 제일 높은 곳이라 우선 정리 대상입니다."

### ② 복용약 점검 — DEP-01 / DEP-02 (의존성 대조, npm/pip 한정)
- 탐지: 선언 목록(package.json의 dependencies / requirements.txt)과 코드의 실제 import·require를 대조.
  - DEP-01: 선언했지만 어디서도 import 안 함 (🟡) — "설치만 해두고 안 먹는 약"
  - DEP-02: import하는데 선언 목록에 없음 (🟡) — "처방전에 없는 약을 먹는 중" (새 컴퓨터에서 설치 누락 사고). 단, **선언 파일 자체가 없으면 DOC-03만 보고하고 DEP-02는 생략** (같은 원인의 이중 보고 금지)
- 네트워크가 필요한 보안 취약점 조회(npm audit 등)는 현재 범위 밖 — 도입 권장 항목으로만 안내.

### ③ 검사 장비 통역 — LINT-01
- 프로젝트에 린터·테스트 설정이 있으면: **코드 실행이므로 경고+승인 필수** ("이 단계는 프로젝트의 코드를 실제로 실행합니다. 출처를 믿을 수 있나요?") → 승인 시 실행하고 결과를 한국어 일상어로 통역해 ⚪ LINT-01로 보고. 거부 시 생략.
- 없으면: 도구 도입을 권장 항목으로 제안 (실행 없음). 이 경우 LINT-01을 **발견ID 블록에 넣지 않는다** (경고 통역이 아니므로 — 권장 텍스트로만).
- 실행으로 생긴 캐시·산출물은 목록을 보고하고, 시나리오 5 검증에서는 예외 처리.

### ④ AI 작업물 문진 — AIGEN-01/02/03 (보수적 판정)
- 문진: "이 프로젝트로 **원래 만들려던 것**을 한 문장으로 알려주실 수 있나요?" (답이 없으면 이 절은 관찰만)
- AIGEN-01 평행 구현: 같은 목적의 기능이 다른 이름·다른 파일로 2벌 존재 (🟡)
- AIGEN-02 과한 추상화: 사용처가 1곳뿐인 추상 계층·인터페이스 (⚪)
- AIGEN-03 미연결 코드: 만들어졌지만 어디서도 연결되지 않은 모듈 — 원래 의도와 대조해 "버려진 시도"로 보이는 것 (🟡)
- **오탐이 많은 영역이므로 확신이 서는 것만 보고하고, 항목마다 판단 근거를 한 줄 덧붙인다.**

### TEST-02 — 깨진 채 방치된 테스트 [정밀]
- 탐지: ③에서 테스트 실행이 승인된 경우, 실패하는 테스트 발견 시.
- 판정: 🟡 권장. / 쉬운 설명: "안전장치(테스트)가 이미 고장 난 채 방치되어 있습니다. 고장 난 화재경보기와 같습니다."

## 검진 절차 (기본 검진)

1. 분석 대상 폴더에서 제외 목록(SKILL.md 절대 규칙 10) 적용 후 파일 목록 작성 — 대상 폴더의 EXPECTED.md 머리말에 "스킬 자체 테스트의 정답지" 표식이 있으면(사본 위치 무관) 그 정답지를 **파일 목록·Grep/Glob 검색 범위·파일 수 집계에서 모두 제외** (절대 규칙 9). 이때 모든 Grep/Glob 호출에 EXPECTED.md 제외 글롭(`!EXPECTED.md`)을 지정해, 검색 결과에 정답이 섞여 들어오는 부분 커닝까지 기계적으로 차단한다
2. **규모 가드**: 제외 목록 적용 후 파일이 100개를 초과하면 — 그대로 진행하지 않고 "프로젝트가 큽니다(N개). ⓐ 코어 11항목(첫 묶음)만 빠르게 검진 ⓑ 하위 폴더를 지정해 범위 축소 중 고르세요"라고 제안한다 (완주 보장이 우선)
3. 기본 검진 항목(인벤토리에서 [정밀] 표시가 없는 ● 22항목 = 코어 11 + 확장 11)을 순서대로 점검 — 각 항목의 탐지 방법대로만. 규모 가드에서 ⓐ를 골랐으면 코어 11항목(본문 첫 묶음)만
4. 발견마다: ID·위치(파일:줄)·심각도 판정 근거를 기록
5. `report-template.md` 양식으로 보고서 작성 (종합 판정·부위별 소견 등급은 아래 규칙)
6. --deep이면 "정밀 검진 절"의 ①~④를 추가 수행

줄 수 측정 기준: **빈 줄을 포함한 전체 줄 수** (PowerShell 기준 `(Get-Content -Encoding UTF8 파일).Count` — PS 5.1은 인코딩 미지정 시 한글 UTF-8 파일의 줄 수를 잘못 셀 수 있으므로 반드시 `-Encoding UTF8` 지정).

## 검진 등급 규칙 (단순 가중 합산)

부위 매핑 — 구조: DUP·BIG·STRUCT / 정리: DEAD·TMP·STALE / 문서: DOC / 테스트: TEST / 안전: HARD(+v1.1: SEC·PII).
부위별 감점: 🔴 1건 = 3점, 🟡 1건 = 1점, ⚪ 1건 = 0.5점 →
**A**: 0점 · **B**: 0.5~2점 · **C**: 2.5~5점 · **D**: 5점 초과. 종합 등급 = 전 영역 감점 합산에 같은 기준 적용 (단, 🔴이 하나라도 있으면 종합은 C를 넘지 못함 — 상한 규칙이므로 합산이 이미 D라면 D 유지).

정밀 전용 항목(HIST·DEP·LINT·AIGEN·TEST-02)은 5개 영역 등급에 넣지 않고 **종합 합산에만** 반영하며, 보고서에 그 사실을 한 줄 명시한다.
