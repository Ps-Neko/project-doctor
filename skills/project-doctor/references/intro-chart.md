# 초진 차트 양식 (intro-chart)

> "내 프로젝트가 뭔지부터 모르겠다" / 인수인계를 주고받을 때. 진단·해설만 하는 **읽기 전용** 모드.
> 통과 기준: **이 차트만 보고(코드를 안 열고) 제3자가 프로젝트 실행까지 도달할 수 있는가.**

## 0. 방향 확인 (질문 1개)

> 이 프로젝트를 **넘겨받으신** 건가요, **넘겨주실** 건가요?
> - 넘겨받음/그냥 궁금함 → **해설 차트** 작성 (1장)
> - 넘겨줄 예정 → 해설 차트 + **인수인계 소견서 옵션** 안내 (2장)

사용자가 이미 상황을 밝혔으면(예: "넘겨받은 프로젝트인데...") 이 질문은 생략하고 바로 진행한다.

## 1. 해설 차트 작성 절차

1. 폴더 확정 (경로 안전 가드·제외 목록 적용 — checkup과 동일)
2. **한 줄 소개**: 엔트리포인트·README·파일 구성으로 "이 프로젝트가 무엇을 하는 것인지" 한 문장으로. 확신이 없으면 "~로 보입니다(추정)"
3. **프로젝트 지도 — 핵심 파일 5개**: 후보 선정 기준 = ① 엔트리포인트(main/index/app) ② 참조가 많은 파일 ③ 큰 파일 ④ 설정 파일. 표로:

```markdown
| 순서 | 파일 | 역할 (일상 언어) |
|------|------|-----------------|
| 1부터 보세요 | main.py | 프로그램의 시작점 — 전체 흐름이 여기 있음 |
```

4. **실행·중지 방법**: README → 패키지 스크립트(package.json 등) → 엔트리포인트 순으로 **정적으로만** 추정 (코드를 실행하지 않는다). 추정이면 반드시 "(추정 — 직접 실행해 확인한 것은 아님)" 표기. 필요한 사전 설치(의존성)도 함께. 비개발자용 주의도 한 줄(예: "Windows 창에서 한글이 깨져 보여도 고장이 아닐 수 있습니다").
5. **만지면 위험한 곳**: 하드코딩 경로·설정 파일·데이터 폴더 등 "잘 모르고 고치면 깨지는" 지점을 **확신 높은 순으로 2~4개** (카탈로그 ID 참조 가능, 추정이면 표기)
6. **미완성·방치 구간**: TODO 무더기, 버려진 실험 파일, 주석 코드 덩어리 등 (STALE/DEAD 항목 재사용)
7. 보고서: report-template 공통 골격 중 **머리말·발견ID 블록은 유지, 검진 결과표는 생략** — 대신 "정밀한 등급은 건강검진(checkup)에서 받을 수 있습니다" 한 줄 (해설 모드는 전수 검진이 아니므로 등급을 매기지 않는다). 본문은 위 2~6 구성 (4요소 대신 — 단, 위험 구역·미완성 구간에서 카탈로그 항목을 인용하면 ID 표기, 인용 ID 없으면 `발견ID: (없음)`)

## 2. 인수인계 소견서 옵션 (넘겨줄 때)

1. **빈칸 진단**: 받는 사람이 물어볼 것들이 문서화되어 있는지 점검 — 설치법 / 실행법 / 설정값 위치와 의미 / 자주 깨지는 곳과 대처 / 미완성 목록
2. 빈칸을 보고서에 나열하고: "승인하시면 누락 문서(인수인계 노트)를 직접 작성해 드립니다"
3. 사용자가 승인하면 — **문서 작성은 쓰기 작업이므로 처방전 프로토콜 경유** (체크포인트 → HANDOVER.md 등 작성 → 전/후 비교 → 되돌리기 한 줄)

## 3. 마무리

- 연계 추천 한 줄: 위험 구역이 많으면 → "건강검진(checkup)으로 정밀 진단을", 공유 직전이면 → "공개 전 검진(release-check)을"
- 진료기록부 운영(record-format.md) 동일 적용
