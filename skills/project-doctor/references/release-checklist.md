# 공개 전 검진 점검표 (release-checklist)

> 회사 공유·GitHub 공개·납품 직전 마지막 점검. 판정은 "자동 점검 통과 / 보류" — 보증이 아니라 점검이다.
> 진단은 읽기 전용. 발견 항목의 수정은 사용자가 승인할 때만 처방전 프로토콜로.

## 0. 검사 제외 선언 (선택)

사용자가 특정 폴더를 "의도된 테스트 자산"으로 선언하면(예: 가짜 비밀키를 일부러 심은 테스트 픽스처), 그 폴더는 검사에서 제외하되 **보고서에 "제외됨 + 사유"를 반드시 표기**한다. 선언 없는 폴더는 모두 검사한다.
- 무관한 폴더가 많은 곳(다운로드 폴더 등)에서는 반대로 **포함 목록**("이 폴더들만 검사해줘")을 받아도 된다 — 보고서에 검사 범위를 명시.
- 읽기 제한 등으로 **내용을 검사할 수 없는 파일**(확장자 정책 밖 텍스트 등)은 이름만 점검하고, 보고서 한계 고지에 "내용 미검사 파일 N개(목록)"를 추가한다.
- 사용자 환경 규칙이 `.env` 접근을 더 강하게 금지하면 **그 규칙이 우선** — 존재 여부만 보고한다.

## 1. 전용 점검 항목 (5종)

### SEC-01 — 하드코딩된 비밀키·토큰 (🔴)
- 탐지 패턴 (Grep, 대소문자 무시):
  - AWS Access Key: `AKIA[0-9A-Z]{16}`
  - GitHub 토큰: `ghp_[A-Za-z0-9]{36}`, `github_pat_[A-Za-z0-9_]{20,}`
  - Slack: `xox[baprs]-[A-Za-z0-9-]{10,}`
  - 개인키 블록: `-----BEGIN [A-Z ]*PRIVATE KEY-----`
  - 일반형: `(api[_-]?key|secret|password|passwd|token)\s*[:=]\s*['"][^'"]{8,}['"]`
  - `.env*` 파일의 `이름=값` 줄 (절대 규칙 4: 파일 내용은 열람하지 않고 **존재와 줄 위치만**)
- 제외(오탐 방지): `YOUR_API_KEY_HERE`·`<...>`·`xxx` 같은 **플레이스홀더 형태**는 보고하지 않음. 공식 예시 키(AWS 공식 문서의 `AKIA…EXAMPLE`형 — 전체 문자열은 자기 검진 매치 방지를 위해 이 문서에 적지 않는다)는 형식이 진짜와 같으므로 **보고하되** "공식 예시 키로 보임 — 확인 필요"를 덧붙인다.
- 검색은 **숨김 파일 포함**으로 — 도구 기본값이 `.env*`·`.gitleaks.toml` 같은 숨김 파일을 건너뛸 수 있으니 숨김 포함 옵션을 명시한다.
- `.env*`의 줄 위치는 값을 읽지 않는 방식으로 파악: `Select-String -Path <파일> -Pattern '^[A-Za-z_]+=' | Select-Object LineNumber` 권장.
- **보고 시 키 내용(값)은 절대 출력하지 않는다** — 파일:줄 + 키 유형만.
- 같은 파일이 여러 항목에 걸리면(예: `.env.backup` = SEC-01 + TMP-01) 각각 보고하되 상호 참조를 한 줄 표기한다.

### SEC-02 — git 과거 기록 속 비밀키 (🔴)
- 전제: 대상에 git이 있을 때만 — 없으면 이 항목은 생략하고 그 사실을 한 줄 보고. 하위 폴더에 별도 git 저장소가 있으면 직접 스캔하지 않고 참고로만 보고.
- 탐지 (값 미출력 방식): `git log --all --oneline -G"<패턴>"` 으로 해당 커밋을 추린 뒤, `git grep -l -E "<패턴>" <커밋>` 으로 파일명만 확인한다 (대소문자 무시는 `(?i)`가 아니라 `-i` 플래그 — git grep은 PCRE `(?i)` 미지원). 이력 스캔 결과에서 정답지(EXPECTED.md) 경로는 제외한다.
- 보고 양식: 4요소의 "승인 명령"은 `(직접 실행하지 않음 — 키 무효화 안내만)` 으로 적는다.
- 발견 시 안내: "과거 기록에서 비밀키 흔적이 발견되었습니다. 기록에서 지우는 작업(이력 세탁)은 전문 작업이라 이 스킬이 직접 하지 않습니다 — 해당 키를 **무효화(재발급)** 하는 것이 가장 확실한 조치입니다."

### PII-01 — 개인정보 (🔴)
- 탐지: 이메일 주소(단, `example.com`·`noreply`·플레이스홀더 제외), 전화번호(`01[016789]-?\d{3,4}-?\d{4}`, 단 `010-0000`류 더미 제외), 주민등록번호 형태(`\d{6}-?[1-4]\d{6}`).
- 보고 시 값은 일부 마스킹(예: `kim.***@***.kr`).

### REL-01 — 라이선스 표기 부재 (🟡)
- 탐지: Glob `LICENSE*`, `COPYING*` 0건 (외부 공개 예정일 때).

### REL-02 — 진료기록부 잔존 (🟡)
- 탐지: `.project-doctor/` 존재. 기록부는 "프로젝트 약점 일람표"이므로 공유 전 제거 권장 — 단 이 검진의 분석 제외 목록과 무관하게 **존재 자체를 보고**한다.

## 2. 기본 카탈로그 재확인 항목 (공개 맥락 — ID 재사용)

- **HARD-01** 내 컴퓨터 전용 경로 (`C:\Users\...` 등) — 받는 사람 PC에서 즉시 깨짐 (🔴)
- **TMP-01** 임시·백업 파일 잔재 — 공개물에 섞이면 안 되는 것 (🟡)
- **DOC-01 / DOC-02** README 부재 / 실행 방법 없음 — "README만 보고 설치~실행" 가능해야 함 (🔴/🟡)

## 3. 판정

- **보류**: 🔴이 1건 이상 → 같은 블록 형식(검진 시점 + 한계 고지 포함)을 쓰되 첫 줄을 "## 판정: 보류 ⛔ — 아래 항목을 해결한 뒤 다시 검진하세요"로
- **자동 점검 통과**: 🔴 0건 → 반드시 아래 형식으로:

```
## 판정: 자동 점검 통과 ✅
- 검진 시점: <일시> (git이면 커밋 해시) · 분석 파일 N개 · 제외 선언: <목록 또는 없음>
- 한계 고지: ⓐ 이 검진은 알려진 패턴만 검사하며 미지 형식은 보장하지 않습니다
  ⓑ 코드 보안 취약점 분석·라이선스 법적 검토는 범위 밖입니다
  ⓒ 판정 이후의 변경에는 효력이 없으며, 최종 공개 책임은 사용자에게 있습니다
```

## 4. 보고서

report-template 공통 골격을 따른다 — 발견 항목은 ID+4요소, 마지막 줄 발견ID 블록, 판정 블록은 "오늘의 처방" 직후에 배치.
- 부위별 소견: 점검 범위 밖 부위(구조·테스트)는 판정 대신 `–(범위 아님)`. 부위 매핑 — 안전=SEC·PII·HARD-01 / 문서=DOC·REL / 정리=TMP-01.
- 재검진 비교(record-format 4장)는 **같은 모드(release-check)의 이전 기록**과만 수행한다.
