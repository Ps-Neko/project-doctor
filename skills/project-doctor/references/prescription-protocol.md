# 실행 처방전 프로토콜 (prescription-protocol)

> 사용자가 보고서에서 항목을 승인했을 때의 실행(치료) 절차. **이 절차 밖에서의 수정은 어떤 경우에도 금지.**
> 핵심 약속: 한 번에 하나씩 · 항상 되돌릴 수 있게 · 승인 안 한 것은 1byte도 건드리지 않는다.

## 0. 실행 게이트 (시작 전 전부 확인)

1. **승인 확인**: 실행 대상은 보고서의 "승인 명령"으로 사용자가 고른 항목 1개. 여러 개를 승인했어도 한 번에 하나씩.
2. **변경 예정 파일 목록 제시**: 실행 전에 "이 항목을 실행하면 이 파일들이 바뀝니다" 목록을 보여주고 진행 확인을 받는다.
3. **사전 질문 (해당 시 반드시)**: 다음이 포함되면 진행 전에 명시적으로 한 번 더 묻는다 —
   파일 **삭제** / **이동** / **이름 변경** / 한 번에 **5개 파일 초과** 변경 / 외부 **패키지 설치**.
4. **쓰기 경계 확인**: 변경 예정 파일이 전부 프로젝트 폴더 안인지 확인. 절대 경로·상위 폴더(`../`) 등 **프로젝트 밖 쓰기는 어떤 이유로도 금지** (백업·기록 포함).
5. **체크포인트 수단 확인**: 아래 1장에 따라 git 또는 백업 폴더 중 하나가 확보되어야 실행 가능. **둘 다 거부되면 실행 모드에 진입하지 않는다** (진단·보고는 계속 가능).

## 1. 체크포인트 (되돌리기 보장)

### 프로젝트에 git이 있을 때
- 미커밋 변경이 있으면 먼저 전체를 체크포인트로 묶는다:
  `git add -A; git commit -m "[주치의] 실행 전 체크포인트"`
- 작업 폴더가 깨끗하면 현재 커밋이 곧 체크포인트다 (해시를 기록해 둔다).

### git이 없을 때 — 사용자에게 두 가지 중 선택을 받는다
- ⓐ **git 시작 안내** (쉬운 말로): "git은 '되돌리기 버튼'을 만들어 주는 도구입니다. 이 폴더에서 한 번만 켜두면 언제든 이전 상태로 돌아갈 수 있습니다. 켜 드릴까요? (`git init` 후 첫 저장)"
- ⓑ **백업 복사**: 변경 예정 파일만 `.project-doctor/backup/<일시-항목ID>/`에 원본 폴더 구조 그대로 복사. 복사 직후 **복원 명령 한 줄**을 만들어 보여준다 — 복원 명령은 **절대 경로**로 생성한다 (어느 폴더에서 실행해도 동작하게).
- ⓐ·ⓑ 둘 다 거부 → "되돌릴 방법이 없어 실행할 수 없습니다"라고 안내하고 실행 중단. 이후 같은 항목을 다시 승인하면 실행 게이트(0장)를 처음부터 다시 밟는다.

## 2. 실행 → 검증 → 보고 (항목 1개 기준)

1. **변경 수행**: 승인 항목의 "고치면?"에 적힌 범위만. 고치는 방법이 여러 가지면 보고서의 "고치면?" 방향을 따르고, 명시가 없으면 가장 적은 파일을 건드리는 방법을 골라 실행 게이트의 파일 목록 제시 때 함께 알린다. 겸사겸사 다른 곳을 고치지 않는다.
2. **검증 — 수단 사다리** (위에서부터, 가능한 첫 번째 것):
   - ① 프로젝트에 테스트가 있으면 실행 — "테스트"란 테스트 프레임워크가 수집하는 것(`tests/` 폴더, `test_*.py`/`*.test.js` 규약, package.json의 test 스크립트)을 말하며, 이름만 테스트 같은 실험 파일은 해당 없음. 프로젝트 코드 실행이므로 **사전 경고+승인 필수**: "검증을 위해 이 프로젝트의 테스트 코드를 실제로 실행합니다. 괜찮은가요?" (한 실행 묶음에서 1회만 물으면 됨)
   - ② 테스트가 없으면 **문법·구조 검사** (프로젝트 로직을 실행하지 않고, **부산물도 남기지 않는** 검사만): Python은 `python -c "import ast,sys; ast.parse(open(sys.argv[1],encoding='utf-8').read())" <파일>` (`py_compile`은 `__pycache__` 부산물을 만들므로 사용했다면 검사 직후 반드시 삭제), JS는 `node --check <파일>`
   - ③ 그것도 불가하면 보고서에 **"자동 검증 불가"를 명시**하고, 되돌리기 명령을 평소보다 눈에 띄게 반복 표기 (진료기록부 도입 버전부터는 기록부에도 보존)
   - **검증 실패 시**: 추가 수정을 시도하기 전에 **체크포인트 복원을 먼저 제안**한다.
3. **변경 확정** (git일 때): 변경 예정 목록의 파일만 지정해 커밋 — `git add <변경한 파일들>; git commit -m "[주치의] <ID> 실행: <한 줄 요약>"`. (`git add -A` 금지 — 검증 부산물 등 미승인 파일이 커밋에 섞이는 것 방지. 단, 1장의 "실행 전 체크포인트"는 전체 스냅샷이 목적이므로 -A가 맞다.) 백업 방식일 때는 별도 확정 단계가 없다 — 백업 폴더 보존이 곧 확정.
4. **전/후 비교 보고** (비개발자 표현):
   - 바뀐 파일 목록 + 무엇이 어떻게 달라졌는지 한 줄씩
   - 검증 결과 (무엇으로 검증했고 결과가 어땠는지)
   - **되돌리기 한 줄 (항상 인쇄)**:
     - git: `git revert --no-edit <변경 커밋 해시>` ("이 한 줄이면 방금 변경만 안전하게 취소됩니다")
     - 백업: `Copy-Item -Recurse -Force ".project-doctor/backup/<일시-항목ID>/*" "<프로젝트 폴더>"` 
5. 다음 승인 항목이 있으면 1번부터 반복 (항목마다 별도 커밋/백업 — 항목 단위로 되돌릴 수 있게).
6. 되돌리기·복원 후에도 `.project-doctor/`(백업·검증 캐시)는 기본적으로 보존한다 (다시 되돌릴 때 필요). 사용자가 정리를 원하면 — 삭제 작업이므로 사전 질문 후 삭제한다.

## 3. 금지 사항 (절대 규칙 재확인)

- 승인 항목의 변경 범위 밖 수정 금지 ("이왕 하는 김에" 금지)
- 실패한 테스트를 지우거나 약화시켜 통과시키기 금지
- 프로젝트 폴더 밖 쓰기 금지
- `git push`, `git reset --hard`, 강제 옵션류는 이 프로토콜에서 사용하지 않는다 (되돌리기는 항상 `revert` 또는 백업 복사)

## 4. 자기 검증 절차 (스킬 품질 측정 전용 — 실사용에선 생략 가능)

> 이 장은 스킬 자체의 품질을 재는 개발·테스트 절차다. 여기서 해시 스냅샷을 프로젝트 밖 임시 폴더에 저장하는 것은 **측정용 예외**로, 절대 규칙 7(실행·백업·기록의 프로젝트 밖 쓰기 금지)에 해당하지 않는다.

### 시나리오 6 검증 (미승인 항목 불변)
1. 실행 직전: `Get-ChildItem -Recurse -File <프로젝트> | Get-FileHash -Algorithm SHA256` 결과를 스냅샷으로 저장 (프로젝트 밖 임시 위치 — 측정용 예외)
2. 실행 직후: 같은 명령 재실행 → 두 스냅샷 비교
3. 통과 기준: 차이가 난 파일 = "변경 예정 파일 목록" + `.project-doctor/` 뿐이어야 함 (검증 부산물이 그 밖의 위치에 생겼다면 실패로 본다)
4. 되돌리기 실행 후: git이면 `git status`·`git diff` 깨끗 — Windows에서는 git의 줄바꿈 자동 변환(autocrlf) 때문에 바이트 해시가 달라질 수 있으므로 **git 환경의 판정은 git 기준**으로 한다. 백업 방식이면 해시가 실행 전 스냅샷과 **바이트 일치**해야 한다.
5. 프로젝트 밖 무변경: 프로젝트의 부모 폴더에 새 파일·폴더가 생기지 않았는지 확인한다 (측정용 스냅샷 파일 제외).

### 시나리오 5 검증 (진단 중 무변경)
- 측정 시점: **진단 종료 직후, 기록부 쓰기 전** 해시 스냅샷을 진단 시작 전 스냅샷과 비교 — 차이 0건 (예외: --deep 코드 실행이 승인된 경우의 캐시 산출물은 목록 보고 후 허용)
