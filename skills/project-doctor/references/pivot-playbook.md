# 방향전환 플레이북 (pivot-playbook)

> "이 프로젝트를 다른 방향으로 바꾸고 싶다" — 안전한 전환 **계획서**를 만드는 모드.
> v1 범위: **계획서까지만.** 실제 실행(코드 변경·이동·폐기)은 사용자가 계획의 항목을 승인했을 때 처방전 프로토콜로, 건강검진 실행과 같은 방식으로만 한다.

## 0. 진입 분기

- 목표가 분명함 ("<목표>"가 주어짐) → 2장 전환 계획으로
- 목표가 없거나 망설임 → 1장 갈림길 비교부터

## 1. 갈림길 비교 (2차 소견)

1. 사용자에게 고민 중인 방향 후보를 **먼저** 묻는다. "없다/모르겠다"는 답을 받은 뒤에만 읽기 전용 현황 분석으로 그럴듯한 후보 2~3개를 제안한다
2. 후보마다 한 표로 비교:

```markdown
| | 방향 A: <이름> | 방향 B: <이름> |
|---|---|---|
| 살릴 수 있는 것 | | |
| 버려야 하는 것 | | |
| 예상 작업량 | 적음/중간/많음 (근거) | |
| 위험 | | |
```

3. **근거 있는 추천** 한 단락 — 어느 쪽이 왜 나은지. 선택이 끝나면 2장으로. 사용자가 선택을 보류하면 비교·추천까지로 보고서를 마감(진료기록부 기재 포함)하고 종료한다 — 나중에 다시 와도 된다.

## 2. 전환 계획 작성 절차

1. **현황 파악**: `.project-doctor/records/`에 최근 checkup 기록이 있으면 재사용, 없으면 구조만 빠르게 파악 (읽기 전용)
2. **유지/수정/폐기 분류** — 주요 구성요소를 **하나도 빠짐없이** 분류한다. 단위 기준: 폴더는 1행으로 묶되 내부 파일의 판정이 갈리면 쪼갠다. README 등 문서 파일도 새 정체성의 영향을 받으면 구성요소에 포함한다:

```markdown
| 구성요소 | 판정 | 근거 (목표 기여도 × 수정 비용) |
|---------|------|------|
| markdown_engine.py | ✅ 유지 | 새 목표에 그대로 필요 + 수정 불필요 |
| themes/ | ❌ 폐기 | 새 목표에 기여 없음 |
| exporter.py | 🔧 수정 | 필요하지만 출력 형식 변경 필요 |
```

   판정 기준: 목표 기여도 높음+수정 비용 낮음=**유지** / 기여도 높음+비용 높음=**수정** / 기여도 낮음=**폐기**
3. **마일스톤 분할 (2~4개)**: 각 마일스톤마다 "여기까지 하고 멈춰도 프로젝트가 동작하는 상태"를 한 줄로 정의한다 — 전환 도중에 그만둬도 망가진 상태로 남지 않게.
4. **폐기 2단계 규칙**: 즉시 삭제 금지 — ① `archive/`로 이동(실행 시 처방전 프로토콜: 이동은 사전 질문 대상) ② 삭제 검토는 **마지막 마일스톤 완료 시**로 계획서에 명기 (역시 사전 질문 후에만 삭제).
5. **전환 완료 체크리스트**: 마지막 마일스톤 뒤에 "전환이 끝났다"를 판정할 조건 3~5개 (예: 새 목표 시나리오 1회 동작, archive/ 정리 결정 완료, README가 새 목표 기준으로 갱신됨)

## 3. 보고서

report-template 공통 골격(머리말·발견ID 블록) + 본문 = 갈림길 비교표(해당 시) → 분류표 → 마일스톤 → 폐기 2단계 → 완료 체크리스트. 검진 결과표는 **checkup 결과를 재사용했을 때만** 포함하고, 그 외에는 생략하고 사유 한 줄 ("등급은 건강검진에서").
분류표가 곧 핵심 산출물 — 주요 구성요소 누락은 실패다. 발견ID 블록은 인용한 카탈로그 ID가 없으면 `발견ID: (없음)`.
